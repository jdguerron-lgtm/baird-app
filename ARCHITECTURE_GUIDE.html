1. Vista General

Baird Service es un marketplace que conecta clientes que necesitan reparar electrodomésticos (lavadoras, neveras, aires acondicionados, etc.) con técnicos verificados en Colombia. Todo el proceso se coordina automáticamente por WhatsApp.

El Problema que Resuelve

Cuando un electrodoméstico se daña, el cliente normalmente tiene que buscar un técnico por su cuenta, sin garantías de calidad ni precios justos. Baird Service centraliza ese proceso: el cliente describe el problema, la app usa inteligencia artificial para hacer un diagnóstico preliminar, y luego notifica automáticamente a técnicos calificados en la zona.

Los 3 Usuarios del Sistema

👤

Cliente

Persona que necesita reparar un electrodoméstico. Llena un formulario web, recibe un diagnóstico IA, y espera a que un técnico acepte el trabajo. Al final confirma si quedó satisfecho.

🔧

Técnico

Profesional verificado que repara electrodomésticos. Recibe notificaciones por WhatsApp, acepta trabajos, sube evidencia fotográfica y firma digital al completar.

🛡️

Administrador

Juan (tú). Gestiona solicitudes, verifica técnicos, sube garantías masivas por Excel, y supervisa todo el flujo desde el panel de administración.

2. Cómo Funciona (Paso a Paso)

Este es el flujo completo desde que un cliente tiene un problema hasta que el servicio se completa:

Flujo Completo del Servicio

1. Cliente llena formulario
→
2. IA diagnostica
→
3. Se notifica a técnicos por WhatsApp
→
4. Primer técnico acepta
4. Técnico asignado
→
5. Técnico realiza servicio
→
6. Sube fotos + firma
→
7. Cliente confirma
→
8. ¡Completado!
Paso 1: El cliente solicita un servicio

El cliente visita baird-app.vercel.app/solicitar y llena un formulario con: nombre, teléfono, dirección, tipo de electrodoméstico, marca, modelo, y una descripción del problema. Si es una garantía, se habilitan campos adicionales.

Mientras el cliente escribe la descripción, la IA (Google Gemini) analiza el texto en tiempo real con un "debounce" (espera medio segundo después de que deja de teclear) y muestra un diagnóstico preliminar: posible causa, estimado de costo, urgencia y partes necesarias.

Paso 2: Se guardan los datos y se notifica

Al enviar el formulario, los datos se validan con Zod (un validador de datos) y se guardan en la base de datos Supabase. Luego el sistema busca técnicos que tengan la especialidad correcta (por ejemplo, si es una lavadora, busca técnicos con especialidad "Lavadoras") y les envía un mensaje de WhatsApp con los detalles del servicio y un botón para aceptar.

Paso 3: El técnico acepta

El primer técnico que toca el botón "Acepto este servicio" en WhatsApp gana la asignación. Es un mecanismo "atómico" — si dos técnicos tocan al mismo tiempo, solo uno queda asignado gracias a una condición en la base de datos (WHERE tecnico_asignado_id IS NULL). El técnico recibe un link a su portal personal donde puede ver los detalles.

Paso 4: El técnico completa el servicio

Después de realizar la reparación, el técnico va a su portal (/tecnico/{token}/completar/{id}) donde debe: subir fotos del antes y después, llenar un checklist de verificación, firmar digitalmente con su dedo, y el GPS captura su ubicación. Todo esto queda como evidencia.

Paso 5: El cliente confirma

El cliente recibe un WhatsApp con un link de confirmación. Al hacer clic, puede decir "Estoy satisfecho" o "Tengo un problema". Si está satisfecho, el servicio se marca como completado. Si reporta un problema, se marca como "en disputa" para que el admin intervenga.

3. Tecnologías Utilizadas

Piensa en estas tecnologías como las herramientas que se usaron para construir la casa:

Next.js 16 Framework

Es el "esqueleto" de la app. Maneja las páginas web, las rutas (URLs), y el servidor. Piensa en él como el arquitecto que organiza todo. Usa el "App Router", que es la forma moderna de crear páginas.

React 19 Interfaz

Es lo que el usuario ve y toca. Los botones, formularios, listas — todo está hecho con componentes de React. Es como los ladrillos individuales de la casa.

TypeScript 5 Lenguaje

El lenguaje de programación. Es JavaScript pero con "tipos" — como poner etiquetas a todo para que el código sea más seguro y tenga menos errores.

Tailwind CSS Estilos

Controla cómo se ve todo: colores, tamaños, espacios, sombras. En lugar de escribir archivos CSS separados, se ponen clases directamente en el HTML como bg-blue-500 text-white p-4.

Supabase Base de Datos

Es donde se guardan todos los datos: solicitudes, técnicos, notificaciones, evidencias. También maneja la autenticación del admin y el almacenamiento de fotos. Es como el "archivador" gigante del negocio.

WhatsApp Business API Mensajería

Permite enviar y recibir mensajes de WhatsApp automáticamente. Cuando un cliente solicita un servicio, el sistema envía WhatsApps a los técnicos sin intervención humana.

Google Gemini AI IA

Inteligencia artificial que analiza la descripción del problema y genera un diagnóstico preliminar: qué podría estar mal, cuánto podría costar, y qué tan urgente es. (Temporalmente desactivado).

Vercel Hosting

Es donde "vive" la app en internet. Cada vez que se sube código nuevo, Vercel automáticamente lo despliega. La app está en baird-app.vercel.app.

4. Arquitectura (Cómo Está Organizado)

La app tiene una arquitectura de "capas", donde cada capa tiene una responsabilidad específica:

🖥️ Capa de Presentación (lo que ve el usuario) Páginas React + Tailwind CSS — formularios, botones, portales
🔀 Capa de Rutas (App Router de Next.js) Decide qué página mostrar según la URL — /solicitar, /admin, /tecnico/[token]
⚙️ Capa de API (endpoints del servidor) 8 endpoints que procesan datos — /api/whatsapp/notify, /api/triaje, etc.
🧠 Capa de Servicios (lógica del negocio) whatsapp.service.ts, solicitud.service.ts — la "inteligencia" de la app
💾 Capa de Datos (almacenamiento) Supabase (PostgreSQL) — 5 tablas + Storage para fotos

6. API Endpoints (8 en total)

Los endpoints son como "puertas de servicio" que reciben peticiones y devuelven respuestas. Están en src/app/api/.

POST /api/whatsapp/notify

Busca técnicos con la especialidad correcta y les envía WhatsApp con los detalles del servicio y botón de aceptación.

POST /api/whatsapp/accept

Procesa la aceptación del técnico. Usa una actualización atómica en la base de datos para evitar que dos técnicos acepten el mismo servicio.

GET/POST /api/whatsapp/webhook

GET: Handshake de verificación con Meta. POST: Recibe eventos de WhatsApp (mensajes, confirmaciones), verifica firma HMAC, y responde 200.

POST /api/triaje

Envía la descripción del problema a Google Gemini AI y devuelve un diagnóstico estructurado en JSON.

POST /api/completar-servicio

Cuando el técnico completa un servicio, este endpoint guarda la evidencia y envía WhatsApp al cliente pidiendo confirmación.

POST /api/confirmar-servicio

Procesa la confirmación del cliente (satisfecho vs. problema) y actualiza el estado de la solicitud.

POST /api/carga-masiva

Recibe un archivo Excel, lo parsea con la librería xlsx, mapea las columnas del formato BITÁCORA, y hace un insert masivo en la base de datos.

GET /api/health

Endpoint de salud que devuelve "OK" — útil para verificar que el servidor está funcionando.

Seguridad del Webhook

Cada mensaje que Meta envía al webhook incluye una firma HMAC-SHA256. La app verifica esta firma comparándola con el WHATSAPP_WEBHOOK_SECRET para asegurar que el mensaje realmente viene de Meta y no de un impostor.

9. Estados de una Solicitud

Cada solicitud pasa por varios estados, como un paquete que se rastrea:

Máquina de Estados

pendiente → notificada → asignada → en_proceso → en_verificacion
en_verificacion → completada ó en_verificacion → en_disputa
En cualquier momento: cancelada

pendiente

El cliente acaba de enviar el formulario. Nadie ha sido notificado todavía.

notificada

Se enviaron WhatsApps a los técnicos compatibles. Esperando que alguno acepte.

asignada

Un técnico aceptó el trabajo. Ya tiene acceso a los detalles en su portal.

en_proceso

El técnico está realizando la reparación (estado intermedio).

en_verificacion

El técnico subió evidencia (fotos, firma). El cliente recibió link para confirmar.

completada

El cliente confirmó que está satisfecho. Servicio terminado exitosamente.

en_disputa

El cliente reportó un problema. Requiere intervención del administrador.

10. Seguridad

Row Level Security (RLS)

Supabase tiene RLS activado en las 5 tablas. Esto significa que la base de datos misma controla quién puede leer y escribir qué datos, independientemente del código de la app.

Autenticación Admin

El panel de administración requiere login con Supabase Auth (email + contraseña). Sin login, no hay acceso a /admin.

Webhook HMAC

Los webhooks de WhatsApp se verifican con firma HMAC-SHA256 para asegurar que realmente vienen de Meta.

Tokens UUID

Los técnicos acceden a su portal con un token UUID único (no su ID). Esto hace imposible adivinar la URL de otro técnico.

Middleware de Seguridad

middleware.ts agrega headers de seguridad a todas las respuestas y maneja rate limiting para evitar abusos.

Validación Zod

Todos los datos del formulario se validan con Zod antes de guardarse. Por ejemplo, el teléfono debe ser un número colombiano válido.
